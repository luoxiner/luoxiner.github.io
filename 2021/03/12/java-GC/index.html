¶分代收集理论

  1. 两个分代理论假说
    1. 弱假说 大多数对象都是朝生熄灭的
    2. 强假说 熬过越多次的垃圾收集过程的对象越难以消亡

根据分代假说将Java内存区域分为不同区域 -> 依据对象的年龄将对象放在不同区域 -> 针对不同区域进行垃圾回收 |-> minor gc , major gc , full gc

​ |-> 不同的垃圾回收算法 -> 标记复制，标记清除，标记整理

但是在进行新生代区域的gc时 （minor gc）新生代的对象可能被老年代所引用，但是将老年代加入gcroot 进行扫描消耗过大 ——————————>>>>>>>>>

第三个经验假说

跨带引用只存在于少数对象之间

因此在这个假说的前提下，我们在进行新生代的垃圾回收时，不用扫描整个老年代，我们只需要维护一个表格记录，老年代的哪些区域引用了新生代，进行扫描时只需要扫描这些区域即可。

¶垃圾回收算法

  1. 标记清除算法

    标记 -> 清除 两阶段

    缺点：

    1. 运行的效率不稳定，当Java堆中有很多对象时，标记清除花费的时间很长。
    2. 标记清除会产生很多内存碎片，容易引发下一次垃圾回收。

  2. 标记复制 （适用于新生代）

    标记 -> 复制 两阶段

    将内存分为两部分，其中一块用完了之后，将这一块中存活的对象复制到另一块中，此时对象之间都是连续的，进行了整理。然后清除用完的内存块，反复进行。

    但是这样内存使用的效率不高，会有一半的内存区域无法使用。

    IBM 公司进行了一项研究表明 98% 的新生代对象熬不过第一次内存回收

    在此基础上我们没必要将内存等分，1:8或者其他比例也可以保证内存复制的时候，空间足够。

    因此将内存分为 eden 和 survivor 两个区域，其中survivor区域分为两部分，每次只是用eden和其中一个survivor区域。

    为什么需要两个survivor区域？

    避免内存配的时候产生更多的碎片

    过程模拟

    进行垃圾回收 ，此时将eden区域的对象复制到survivor区域，当进行下一次回收的时候survivor区域和eden区域都有对象此时不能找到一块空闲的区域进行复制，因此需要两块survivor区域。

    在survivor区域无法存放一次gc产生的存储或对象时该怎么办？

    依赖其他区域进行担保的策略，通常是老年代进行担保。

  3. 标记整理算法

    标记 -> 整理

    整理的时候讲存活对象移动到内存的一端，然后清理边界。

    缺点：移动较多对象并且重新更新引用是巨大的消耗。

    标记整理和标记删除两种算法在标记完对象后针对存活对象是否移动。

    移动会产生巨大消耗，产生“stop the world” ，如果不移动就会产生更多的内存碎片。

    可以使用一个这种的方法，在内存不是很紧张的时候使用标记清理，这时只需要很短的停顿时间，在内存紧张的时候使用标记整理。

¶算法细节

¶根节点枚举

由于gcroot在全局性的引用和应用的上下文中，如果对所有的对象进行扫描，花费很大，HotSpot虚拟机采用空间换取时间的策略，维护一个OopMap（Ordinary Object Pointer Map）数据结构存储引用的位置。我们只需要在这些位置中开始可达性分析即可。

有关如何确定类中各个引用的位置在后续介绍

¶安全点

安全点的选定规则：是程序能够长时间运行为特征 例如循环，函数调用，异常跳转等需要代码复用的地方。

为什么需要安全点？

虚拟机应该不会为每一条指令都产生Oop，无论时间还是空间消耗太大，因此应该有些Oop是冗余的，我们只需要在一些特定的地方记录这些信息即可，在代码进行复用的时候。

然后进行 gc 操作，此时Java堆中的变化较小，进行垃圾回收较为安全。

但是不是在进行gc的时候每一个线程都能跑到安全点，如何操作？

抢先式中断和主动式中断

抢先式中断在gc时直接中断线程，如果线程没有到达安全点，则让线程继续执行。

主动式中断是设置一个中断标志，线程不断轮询自己的中断标志，在到达终端点的时候自己中断挂起。

常采用主动式中断，轮询频率很高，因此需要很高效，hotspot中将0x106100页设置为不可读，当轮询进行test时会因为缺页挂起，并使用预先注册的异常处理器进行处理。

¶安全区域

考虑到挂起线程无法到达安全点，我们引入安全区域。

定义为程序处于安全区域时，引用关系不会发生变化。

¶记忆集和卡表

前文已说明在进行部分区域的gc时存在跨代引用，我们使用记忆集来存储跨代引用。

记忆集的实现分为不同尺度

  1. 直接记录引用。
  2. 记录存储引用的对象
  3. 记录存储引用的卡片

卡表是记忆集的一种是先形式，将可用内存划分为大小相等的卡片，使用卡表来管理每个卡片中是否存在跨代引用。

¶写屏障

在引用赋值的时候需要改变卡表的的dirty状态，使用写屏障来实现，写屏障是面向引用赋值这个动作的切面，分为写前屏障和写后屏障。

在赋值之后，进行卡片的状态变更，这些代码是由虚拟机自动生成的。

¶并发的可达性分析

在进行可达性分析的时候，如果不是并发进行会使得垃圾回收停顿较长时间，我们考虑并发模式下的可达性分析。

在并发的状态下，以下情况导致可达性分析的结果不准确。

一种是灰色节点指向的引用被删去，对象被黑色节点所引用。

因此不安全状态存在一个充分必要条件

赋值器插入了一条或多条从黑色对象到白色对象的新引用

赋值器删除了全部从灰色对象到该白色对象的直接或间接引用

针对这两种情况，我们只需要破坏其中一条即可记性并发的可达性分析。

  1. 在分析时，如果产生灰色节点的孩子被赋值给黑色节点，将黑色节点记录，并发扫描结束后在对这些记录的节点在进行扫描。这种方法叫做增量更新
  2. 原始快照方法。无论删除与否，灰色节点所指的白色节点的引用关系都会被记录，在并发扫描结束之后，会对这些关系重新扫描，因此看来，无论删除与否，都是对扫描开始的的那一刻的对象关系图进行的。

¶典型垃圾收集器

¶Serial 收集器

采用标记复制的回收算法。

在进行标记复制的时候需要挂起用户线程产生“stop the world” .

优点是，占用的资源少，在客户端程序中尤为何时，因为客户端的新生代区域的大小一般不是很大，收集一次只需要十几或者几十毫秒的时间。对于单核心的机器来说是很好的选择。

Old版本使用标记整理的算法

¶ParNew

这是Serial收集器的并发版本，在进行标记复制的过程中进行了并发处理。

和CMS进行配合

¶Parallel Scavenge收集器

同样是基于标记复制算法，支持并发。

前面两个收集器更多的关注于垃圾回收所带来的停顿，而这款则更加关注吞吐量。

允许用户指定一个最大停顿时间，回收器尽量保证在这个时间段内的停顿。

收集器通过调整新生代的大小来完成停顿时间的调控，因此更短的停顿时间，意味着更小的新生代大小以及更小的吞吐率，更多次的垃圾手机操作。

Old版本使用标记整理的版本

¶CMS收集器

Concurrent Mark Sweep

并发的给予标记清除的算法的收集器。

四个阶段

graph LR
A[初始标记] --> B[并发标记] --> C[重新标记] --> D[并发清除]

其中初始标记是枚举GCroot，需要短暂的停顿，并发标记和重新标记是指前文所说的增量标记。

重新标记也需要短暂的停顿。

优点：

实现了并发的标记，降低了停顿时间。

缺点：

产生内存碎片，需要full gc进行碎片整理

对处理资源十分敏感，虽然并发不会导致用户线程的停顿，但是会降低用户线程的吞吐率。

在处理器资源不多的情况下，会占用较多资源进行垃圾回收。

在进行并发标记和并发清除的时候会产生新的垃圾，这部分垃圾需要等到下一次垃圾回收才能清除，这部分垃圾成为浮动垃圾。

同样并发的时候程序需运行需要空间，因此不能等到老年代完全填满的时候在进行垃圾回收。

¶G1

G1是跨时代意义的垃圾回收器，使用region而不是分代来管理内存。

将内存划分为很多个region，把不同数量的连续的region划分为不同的代进行管理。

关于跨region的引用和跨带引用处理思路类似，但是更加复杂，使用一个双向映射，key

是其他region的起始地址，value是卡表的索引号的集合。

同样在进行并发处理的时候，需要一部分区域给用户线程new对象，region中划分一部分区域作为新对象区域，使用两个指针来管理此部分。

依然使用四个阶段

采用标记整理的算法，在处理并发标记中的准确性时采用原始快照的方法

筛选回收会根据每一块region的回收效益对region’进行排名，从而在指定的时间内回收垃圾。

但是不能一味地把停顿时间设置的太小，导致新产生的对象的速度大于收集垃圾的速度，导致对重的full gc

有用于存储大对象的humongous region

缺点

占用很大的内存，因为G1需要维护很多region 回收信息和跨region引用信息。

¶总结

低延时收集器的几个指标：

  1. 占用的内存大小
  2. 吞吐量
  3. 延迟

¶Shenandoah

和G1的不同点：

  • 在描述不同region之间的引用关系的时候使用一个二维的链接矩阵来描述，更加节省内存，并且降低了伪共享的概率

工作分为九个阶段

  • 现在回收集中再无存货对象，对回收集进行清理

转发指针

存在并发安全问题

1）收集器线程复制了新的对象副本；

2）用户线程更新对象的某个字段；

3）收集器线程更新转发指针的引用值为新副本地址。

复制对象中的对应的值没被更新

通过CAS操作保证并发访问的正确性

众多的读屏障会带来很大的消耗，修改为引用型的读屏障，只有引用类型的读取才会进行相关操作。
